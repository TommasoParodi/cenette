# Cenette — Database & Supabase Architecture

## Overview

Cenette uses Supabase as:

- Postgres database
- Authentication (Google OAuth)
- Row Level Security (RLS)
- Storage (private buckets)

All security logic is enforced at DB level via RLS.
The frontend must never bypass these constraints.

---

# Auth & Profiles

## Table: profiles

Linked 1:1 with auth.users.

Columns:
- id (uuid, PK, FK -> auth.users.id)
- display_name (text)
- avatar_url (text)
- created_at (timestamptz)

Trigger:
- handle_new_user()
  - Runs AFTER INSERT on auth.users
  - Creates row in profiles
  - On conflict: does nothing

RLS:
- Users can SELECT only their own profile
- Users can UPDATE only their own profile

Future:
- profile_photo_path (for custom avatars)
- bio / username

# Groups

## Table: groups

Columns:
- id (uuid, PK)
- name (text)
- vote_mode (group_vote_mode)
- invite_code (text, unique, auto-generated 8 char)
- created_by (uuid, FK -> profiles.id)
- created_at

Invite code generated by:
- generate_invite_code()

RLS:
- SELECT: only if user is group member
- INSERT: only if created_by = auth.uid()

---

## Table: group_members

Columns:
- group_id (uuid, FK -> groups.id)
- user_id (uuid, FK -> profiles.id)
- joined_at
- PK (group_id, user_id)

RLS:
- SELECT: only members of group
- INSERT: only self
- DELETE: only self

Helper:
- is_group_member(group_id)

---

# Entries (Events)

## Table: entries

Columns:
- id
- group_id
- type (HOME | OUT)
- title
- description
- happened_at (date)
- created_by
- created_at
- updated_at

Trigger:
- set_updated_at()

RLS:
- SELECT: group members
- INSERT: group member AND created_by = auth.uid()
- UPDATE/DELETE: only entry creator

Helper:
- is_entry_creator(entry_id)

---

# Entry Photos

## Table: entry_photos

Columns:
- id
- entry_id
- storage_path
- created_at

Constraint:
- Max 3 photos per entry (DB trigger)

RLS:
- SELECT: group members
- INSERT/DELETE: only entry creator

Storage bucket:
- entry-photos (private)

Path convention:
entries/<entry_id>/<file>

Columns:

- PK (entry_id, user_id)

RLS:
- SELECT: group members
- INSERT/DELETE: only entry creator

# Reviews

## Table: reviews

Columns:
- id
- entry_id
- user_id
- rating_overall (1–10)
- comment
- rating_cost
- rating_service
- rating_food
- rating_location
- photo_path
- created_at
- updated_at

Constraints:
- UNIQUE(entry_id, user_id) → max 1 review per user

RLS:
- SELECT: group members
- INSERT: group member AND user_id = auth.uid()
- UPDATE/DELETE: only review author

Helper:
- is_review_author(review_id)

Storage bucket:
- review-photos (private)

Path convention:
reviews/<review_id>/<file>

---

# Avatars (Future Profile Editing)

Storage bucket:
- avatars (private)

Path convention:
users/<user_id>/avatar.jpg

RLS:
- INSERT/DELETE: only owner
- SELECT: only users who share at least one group

Helper:
- shares_group_with(user_id)

# RPC Functions (Atomic Business Logic)

## rpc_create_group(name, vote_mode)

Creates:
- groups row
- group_members row (self)

Returns:
- created group

---

## rpc_join_group(invite_code)

- Finds group by invite_code
- Inserts membership (if not exists)

Security:
- security definer
- requires auth.uid()

---

# Security Model

All data access is enforced via:

- RLS
- auth.uid()
- helper functions

Frontend must never assume permissions.
If something fails, check RLS first.

---

# What Agents Should NOT Modify

- RLS policies
- Helper SQL functions
- Storage path conventions
- RPC function signatures

Changes here must be coordinated.
