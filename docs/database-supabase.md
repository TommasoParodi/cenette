# Cenette — Database & Supabase Architecture

## Overview

Cenette uses Supabase as:

- Postgres database
- Authentication (Google OAuth)
- Row Level Security (RLS)
- Storage (private buckets)

All security logic is enforced at DB level via RLS.
The frontend must never bypass these constraints.

### RLS: regola importante

Con RLS **attivo** su una tabella, ogni operazione (SELECT, INSERT, UPDATE, DELETE) è consentita **solo** se esiste una policy che la permette.

- **Se manca la policy** per un comando (es. UPDATE o DELETE), il comando **non fallisce** ma **nessuna riga viene toccata** (0 righe aggiornate/eliminate).
- Il DB può comunque rispondere "OK" (es. 204 per DELETE), ma le righe non cambiano.

Quindi: per ogni tabella con RLS, assicurati di avere policy esplicite per **tutte** le operazioni che l’app deve fare (SELECT, INSERT, UPDATE, DELETE). Senza policy UPDATE/DELETE, modifica ed eliminazione non avranno effetto.

# Auth & Profiles

## Table: profiles

Linked 1:1 with auth.users.

Columns:
- id (uuid, PK, FK -> auth.users.id)
- display_name (text)
- avatar_url (text)
- created_at (timestamptz)

Triggers:
- handle_new_user()
  - Runs AFTER INSERT on auth.users
  - Creates row in profiles
  - On conflict: does nothing
- on_profiles_delete_delete_auth_user (vedi [supabase-profile-delete-cascade-auth.md](supabase-profile-delete-cascade-auth.md))
  - Runs AFTER DELETE on profiles
  - Deletes corresponding row in auth.users so the user can no longer log in

RLS:
- Users can SELECT only their own profile
- Users can SELECT profiles of users who share at least one group (per mostrare nomi/iniziali in gruppo e eventi); vedi [supabase-profiles-select-group-members.md](supabase-profiles-select-group-members.md)
- Users can UPDATE only their own profile

Future:
- profile_photo_path (for custom avatars)
- bio / username

# Groups

## Table: groups

Columns:
- id (uuid, PK)
- name (text)
- invite_code (text, unique, auto-generated 8 char)
- created_by (uuid, nullable, FK -> auth.users.id ON DELETE SET NULL)
- created_at

Invite code generated by:
- generate_invite_code()

RLS:
- SELECT: only if user is group member
- INSERT: only if created_by = auth.uid()

---

## Table: group_members

Columns:
- group_id (uuid, FK -> groups.id)
- user_id (uuid, FK -> profiles.id)
- joined_at
- PK (group_id, user_id)

RLS:
- SELECT: only members of group
- INSERT: only self
- DELETE: only self

Helper:
- is_group_member(group_id)

# Entry Participants (chi può scrivere una recensione)

## Table: entry_participants

Collegamento evento ↔ membri che hanno partecipato. Solo i partecipanti possono scrivere una recensione; l’evento è visibile a tutti i membri del gruppo. I partecipanti li sceglie (e può modificare) il creatore dell’evento.

Columns:
- entry_id (uuid, FK -> entries.id)
- user_id (uuid, FK -> profiles.id)
- PK (entry_id, user_id)

RLS:
- SELECT: membri del gruppo
- INSERT/DELETE: solo creatore dell’entry

Helper:
- is_entry_participant(entry_id)
- entry_has_participants(entry_id)

Trigger:
- **on_entry_participant_deleted** (AFTER DELETE su entry_participants): chiama `delete_review_when_participant_removed()` e cancella la recensione di quel (entry_id, user_id). Così, se il creatore rimuove un partecipante, la sua recensione viene eliminata automaticamente.

# Reviews

## Table: reviews

Columns:
- id
- entry_id
- user_id
- rating_overall (1–10)
- comment
- rating_cost, rating_service, rating_food, rating_location (1–10, nullable; usati quando l’evento ha vote_mode DETAILED)
- photo_path
- created_at
- updated_at

Constraints:
- UNIQUE(entry_id, user_id) → max 1 review per user

RLS:
- SELECT: group members
- INSERT: group member AND user_id = auth.uid() AND (is_entry_participant(entry_id) OR NOT entry_has_participants(entry_id))
- UPDATE/DELETE: only review author OR entry creator (per eliminare tutte le recensioni al cambio modalità voto)

Helper:
- is_review_author(review_id)

Storage bucket:
- review-photos (private)

Path convention:
reviews/<review_id>/<file>

# Comportamento alla cancellazione del gruppo

Quando il creatore elimina un gruppo (`DELETE` su `groups`), Postgres elimina in cascata (grazie a **ON DELETE CASCADE** sulle FK):

- **group_members**: tutte le membership del gruppo
- **entries**: tutti gli eventi del gruppo
- Per ogni entry eliminata: **reviews**, **entry_cooks**, **entry_participants**, **entry_photos**

Quindi eliminando il gruppo si eliminano anche eventi e recensioni (e il resto dei dati collegati), senza doverli cancellare dall’app. Le FK coinvolte sono: `entries.group_id`, `reviews.entry_id`, `entry_cooks.entry_id`, `entry_participants.entry_id`, `entry_photos.entry_id`, `group_members.group_id`. Per verificare le regole attuali: SQL Editor → query “Foreign key e regole di delete” in [supabase-export-schema-queries.md](supabase-export-schema-queries.md).

# Security Model

All data access is enforced via:

- RLS
- auth.uid()
- helper functions

Frontend must never assume permissions.
If something fails, check RLS first.

# What Agents Should NOT Modify

- RLS policies
- Helper SQL functions
- Storage path conventions
- RPC function signatures

Changes here must be coordinated.
